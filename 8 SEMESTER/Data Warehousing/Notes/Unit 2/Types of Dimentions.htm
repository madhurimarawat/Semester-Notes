Data Warehouse Dimensions 📊

Properties 📜 Junk Dimension 🗑️ Conformed Dimension 🔄 Slowly Changing Dimension 🕰️ Degenerate Dimension 🚫
Definition 📖 Groups small, unrelated attributes into one table to reduce clutter. Used consistently across multiple fact tables for uniform data. Tracks changes in data over time to maintain history. A dimension key stored inside a fact table without a separate dimension table.
Example 🎯 Flags, statuses like "VIP Customer" or "Fraud Alert." Customer details, product info, time dimensions. Tracking address changes or job role history for an employee. Order ID or Invoice Number inside a sales transaction table.
Purpose 🎯 Combines minor attributes to improve database efficiency. Ensures consistency across multiple reports and data warehouses. Preserves historical changes for tracking and reporting. Reduces table joins and speeds up queries.
Storage 💾 Stored as a single compact table. Used across multiple fact tables. Maintains historical versions of data. Stored directly inside the fact table.
Complexity 🧩 Low - Simple and easy to maintain. Medium - Requires careful design to ensure consistency. High - Needs strategies to handle historical changes. Low - No separate table required.
Real-World Example 🌍 A call center tracking customer preferences:
  • "Do Not Call" flag (1 = Yes, 0 = No) 📵
  • "Preferred Language" (1 = English, 2 = Spanish, 3 = French) 🌎
 A retail store and bank sharing the same customer data:
  • Customer ID: 12345 🆔
  • Name: John Doe 👤
  • Used in banking and shopping records 🏦🛍️
 Employee promotion history in HR systems:
  • Employee ID: 7890 🏢
  • Job History: Junior Engineer → Senior Engineer → Manager 📈
  • Tracks role changes over time ⏳
  • Sales order records in an e-commerce database:
  • Order ID: 56789 🛒
  • Product: Laptop 💻
  • Price: $1200 💰
  • Order ID stored inside the fact table 📊
